# 구조맵 Draft 상태 갱신 작업 로그  
(질문 → 문제 인식 → 해결 과정 기록)

---

## 1. 나의 질문  
**“구조맵 생성이 비동기 작업인데, 사용자는 지금 무슨 상태인지 어떻게 알 수 있을까?”**

### 문제 인식  
- 구조맵 생성 요청 후 바로 Draft 카드가 생성되지만  
- 작업이 끝나도 카드 상태가 자동으로 바뀌지 않음
- 새로고침을 해야만 `processing → done` 상태를 확인할 수 있었음

### 해결 방향  
- Draft 상태를 **주기적으로 확인해서 자동 갱신**해야 한다고 판단
- 단, 무조건 실시간으로 처리하는 방식은 피하고 싶었음

---

## 2. 나의 질문  
**“DraftMapCard 컴포넌트 안에서 상태를 직접 갱신하면 안 될까?”**

### 문제 인식  
- 카드 단위로 상태 조회를 하게 되면:
  - 카드 개수만큼 API 요청 발생
  - 관리 포인트가 분산됨
  - 렌더링 구조가 복잡해짐
- 구조적으로 **잘못된 책임 분리**라는 판단

### 해결 방향  
- DraftMapCard는 **표시 전용 컴포넌트**
- 상태 갱신은 반드시 **부모 페이지에서 리스트 단위로 처리**하기로 결정

---

## 3. 나의 질문  
**“그러면 Draft 리스트를 한 번에 계속 polling 하면 되지 않을까?”**

### 문제 인식  
- Draft가 모두 완료된 이후에도 polling이 계속된다면:
  - 불필요한 네트워크 요청
  - 비용 및 성능 낭비
- Supabase Realtime을 쓰자니:
  - 비용이 발생
  - 이 정도 실시간성이 꼭 필요한 상황은 아님

### 해결 방향  
- Draft 중 **`processing` 상태가 하나라도 있을 때만** polling
- 모두 완료되면 **자동으로 polling 중단**
- 실시간성보다 **조건부 상태 확인**에 초점

---

## 4. 나의 질문  
**“이 상태 조회 로직, useSWR로 처리하는 게 맞을까?”**

### 문제 인식  
- 과거에는 notification 폴링에서 React Query를 추천받아 사용한 경험이 있음
- 이번 케이스는:
  - 단순 조회
  - 조건부 polling
  - 캐시 + focus revalidate 정도면 충분

### 해결 방향  
- `useSWR`의 특성이 이번 요구사항에 더 잘 맞는다고 판단
  - key가 null이면 자동 비활성화
  - `refreshInterval`을 조건부로 제어 가능
- Draft 상태 조회는 `useSWR`로 구현

---

## 5. 나의 질문  
**“이 로직을 페이지에 그대로 두는 게 맞을까?”**

### 문제 인식  
- 페이지 코드가 점점 길어지고 있음
- Draft 상태 동기화 로직은:
  - UI와 직접 관련 없음
  - 다른 페이지에서도 재사용 가능

### 해결 방향  
- 상태 polling + merge 로직을  
  **`useMapDraftStatusPolling`이라는 custom hook으로 분리**
- 페이지는:
  - Draft state만 관리
  - hook을 호출해서 결과만 반영

---

## 6. 나의 질문  
**“서버에서 내려온 상태를 어떻게 Draft에 안전하게 반영할까?”**

### 문제 인식  
- 매 polling마다 무조건 state를 바꾸면:
  - 불필요한 re-render 발생
- Draft 리스트가 커질 경우 성능 문제 가능

### 해결 방향  
- 서버 응답을 `Map(id → row)` 형태로 변환
- 기존 Draft와 비교해서 **실제 변경이 있는 경우에만 갱신**
- thumbnail은 변경 시 cache buster 적용

---

## 최종 결과

- DraftMapCard는 아무 로직 없이 표시만 담당
- 부모 페이지에서 Draft 리스트 단위로 상태 관리
- `processing` 상태가 있을 때만 polling
- 모든 Draft 완료 시 polling 자동 중단
- 새로고침 없이 상태 변화 즉시 반영

---

## 이번 작업으로 정리된 기준

- 비동기 작업 상태 관리는 **카드 단위 ❌ / 리스트 단위 ⭕**
- 실시간이 필요 없는 경우 Realtime ❌
- 조건부 polling + 명시적 상태 관리 ⭕
- 페이지 비대화 방지를 위한 hook 분리 ⭕
