# LeftPanel 레이아웃 리팩터링 작업 로그

## 작업 배경
LeftPanel에서 맵(Map) 상세 정보를 보여주는 UI를 정리하는 과정에서  
정보의 우선순위와 가독성을 다시 조정할 필요가 있었다.  
특히 description과 태그의 위치 및 표현 방식이 사용 흐름에 맞지 않았다.

---

## 질문 1  
description을 접었다 폈다 하는 UI가 필요한가?

### 문제 인식
- description은 길어질 수 있지만,  
  해당 맵을 이해하는 핵심 정보임에도  
  기본적으로 숨겨져 있어 접근성이 떨어짐
- 매번 “설명 보기” 버튼을 눌러야 하는 UX가 번거로움
- 정보 패널 성격상 **항상 노출되는 편이 자연스러움**

### 문제 해결
- description의 접기/펼치기 로직 완전 제거
  - `useState(descOpen)` 삭제
  - 토글 버튼 제거
- description을 항상 표시하도록 변경
- 긴 텍스트도 자연스럽게 읽히도록
  - `whitespace-pre-wrap`
  - `break-words` 유지

### 결과
- 정보 패널의 성격이 명확해짐
- “읽기 위해 클릭해야 하는 정보”가 사라져 UX 개선
- description이 핵심 정보로 인식됨

---

## 질문 2  
태그와 description 중 어떤 정보를 먼저 보여주는 것이 맞을까?

### 문제 인식
- 기존 구조에서는 description이 태그보다 위에 위치
- 하지만 실제 사용 흐름에서는:
  - 태그 → 이 맵의 성격을 빠르게 파악
  - description → 상세 맥락 이해
- 정보 스캔 순서가 직관적이지 않았음

### 문제 해결
- 섹션 순서 변경
  - `태그 섹션`을 `description 섹션` 위로 이동
- UI 컴포넌트 구조는 그대로 유지하면서
  - 렌더링 순서만 조정

### 결과
- 패널을 열자마자:
  1. 출처
  2. 태그 (구조화 정보)
  3. 설명 (상세 정보)
- 정보 인지 흐름이 자연스러워짐
- “한눈에 파악 → 자세히 읽기” 구조 완성

---

## 최종 정리
이번 작업은 기능 추가가 아닌,  
**정보의 위계와 읽는 순서를 바로잡는 리팩터링**이었다.

- 불필요한 인터랙션 제거
- 핵심 정보의 항상 노출
- 정보 스캔 흐름 개선

LeftPanel이  
“컨트롤 UI”가 아니라  
**읽는 정보 패널**이라는 성격을 더 명확히 하게 되었다.
