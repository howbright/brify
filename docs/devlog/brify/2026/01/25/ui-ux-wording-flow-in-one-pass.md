# UI · UX · 워딩 · 화면설계, 한 번에 다듬는 실제 작업

이 문서는 Brify 서비스의 실제 작업 과정을 기록한 문서다.  
기획 → 화면설계 → 워딩 → UI → UX를 **단계별로 나누지 않고**,  
하나의 흐름 안에서 동시에 다듬는 방식을 그대로 남긴다.

---

## 이 작업의 전제

- UX는 화면이 아니라 **흐름**이다
- 워딩은 장식이 아니라 **기능**이다
- UI는 디자인이 아니라 **의사결정의 결과**다

그래서 이 작업에서는  
“기획 먼저 / 디자인 나중” 같은 구분을 하지 않는다.

---

## 작업 흐름

### 1. 기능이 아니라 사용자 상태부터 본다
- 지금 사용자는 무엇을 하려는 상태인가?
- 망설이는 지점은 어디인가?
- 되돌릴 수 없는 선택은 무엇인가?

→ 이 질문이 버튼, 문구, 다이얼로그 구조를 결정한다.

---

### 2. 워딩을 가장 먼저 고친다
- `생성하기` → `구조맵 생성하기`
- `다른 콘텐츠 생성` → `다른 구조맵 생성하기`
- 취소 버튼을 없애고  
  → “이 작업은 되돌릴 수 없습니다”라는 **사전 고지**로 대체

워딩을 바꾸면  
UX와 UI 구조가 자동으로 따라 바뀐다.

---

### 3. 모바일에서 먼저 깨본다
- 긴 제목에서 버튼이 구겨지는 문제
- 날짜 + 버튼이 한 줄에 들어가 답답해지는 문제
- 다이얼로그에서 가로 스크롤이 생기는 문제

→ 전부 **레이아웃 문제가 아니라 UX 문제**로 보고 수정

---

### 4. 다크모드는 ‘분위기’가 아니라 ‘구분감’
- 다크모드에서 카드와 배경이 섞이면 UX는 무너진다
- 밝기 차이 + ring + shadow로 **레이어 구분**을 명확히

---

### 5. 자동 기능은 ‘있어 보이게’ 만들지 않는다
- 자동 키워드 추출 → 실제 사용성 낮음 → 제거
- 태그는 사용자가 직접 입력하는 구조로 단순화
- “유튜브 자동 추출”도 과장 없이, 현재는 모킹임을 인지

---

## 이 방식의 핵심

- UI / UX / 워딩 / 기획을 **분리해서 생각하지 않는다**
- 화면을 보면서 동시에 결정한다
- 실제 서비스 코드 기준으로 판단한다
- 예쁜 것보다 **덜 헷갈리는 것**을 우선한다

---

## 이 문서의 목적

- 완성된 결과를 설명하기 위함이 아니다
- “어떻게 생각하면서 만들었는지”를 남기기 위함이다
- 1인 개발, 사이드 프로젝트, 실제 서비스 개발자에게
  가장 현실적인 방식이라고 믿는다
