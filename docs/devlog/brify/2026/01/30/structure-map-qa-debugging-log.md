
### 원인 분석
- util 파일 최상단에서 OpenAI client를 즉시 생성
- NestJS 부트스트랩 이전에 env가 로드되지 않은 상태에서 참조됨

### 해결 방식
- OpenAI client를 **ConfigService 기반으로 주입**
- util 함수 → service 형태로 변경
- 환경변수 접근 시점을 런타임으로 지연

이 문제를 통해  
**“env가 있다”와 “언제 읽히는가”는 완전히 다른 문제**라는 점을 다시 확인했습니다.

## 3. 세 번째 문제: Queue는 동작하는데 상태를 볼 수 없음

### 발생한 문제
- 크레딧 차감 이후 job은 정상적으로 enqueue 되는 것 같았음
- 하지만 Redis 상태를 눈으로 확인할 수단이 없음
- 예전에 봤던 대시보드가 기억나지 않음

### 접근 방식
- Redis GUI를 먼저 의심 (RedisInsight)
- Docker 컨테이너 목록 확인
- 실제로 필요한 것은 **Redis 자체 상태가 아니라 Queue 상태**라는 점을 재인식

### 해결
- 이전에 사용했던 것이 **Bull Board**라는 것을 확인
- NestJS API 서버에 Bull Board를 다시 연결
- `/admin/queues` 경로로 접근 가능하게 설정

## 4. 네 번째 문제: Bull Board에 큐가 일부만 보임

### 발생한 문제
Bull Board 페이지는 열리지만,
- 일부 큐만 보이고
- 구조맵 생성 큐가 보이지 않음

### 원인 분석
- Bull Board는 **현재 프로세스에 등록된 큐만** 표시
- `MapsModule`에서 `BullModule.registerQueue()`는 했지만
- Bull Board 쪽에서 해당 큐를 명시적으로 추가하지 않음

### 해결 방식
- `getQueueToken()`을 사용해 큐 인스턴스를 직접 가져옴
- 큐 이름 배열을 만들어 여러 큐를 한 번에 등록
- 존재하지 않는 큐는 warn 로그만 남기고 skip 처리

이를 통해  
**API 서버 기준으로 등록된 모든 큐를 한 화면에서 확인 가능**해졌습니다.

## 5. 현재 상태 구조화

현재 구조맵 생성 로직은 다음 흐름으로 안정화되었습니다.

1. 클라이언트에서 구조맵 생성 요청
2. 서버에서 입력 텍스트 기준 크레딧 계산
3. 트랜잭션으로 크레딧 차감
4. Bull Queue에 구조맵 생성 작업 적재
5. Bull Board를 통해 job 상태 실시간 확인 가능
