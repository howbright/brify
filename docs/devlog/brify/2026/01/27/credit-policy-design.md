# 크레딧 정책 설계 & 협업 방식 정리 (Devlog)

> 이 문서는 실제 개발 중 **요구사항 협의 → 정책 확정 → UI 문구 정리 → 다국어 적용**까지의 과정을 기록한 md입니다.

---

## 1. 문제 배경

* 텍스트/스크립트 분량에 따라 크레딧이 차감되는 기능을 구현해야 했음
* 기존 기준은 불명확했고, 사용자에게 **예측 가능하고 공정한 과금 기준**이 필요했음

---

## 2. 최종 확정된 크레딧 정책

### 📏 과금 기준 분량 (정규화 후 글자 수 기준)

* **70,000자 이하** → **1 크레딧**
* **70,001 ~ 110,000자** → **2 크레딧**
* **110,000자 초과** → **처리 불가 (요청 거절)**

> 핵심 메시지: **대부분의 콘텐츠는 1크레딧만 사용**

---

## 3. 정규화 기준 (Billing Normalization)

과금 기준 글자 수는 원문이 아니라, 아래 정규화 후 텍스트를 기준으로 계산함.

### 제거 대상

* 타임스탬프

  * `0:03`, `12:34`, `1:02:11`
  * `0:03 내용...` 형태 포함
* 자막 태그

  * `[음악]`, `[박수]`, `[기도]`, `[BGM]`, `[Music]`, `[Applause]` 등

### 정리 규칙

* 연속 공백 → 단일 공백
* 과도한 줄바꿈 정리

> 목적: **실제 의미 있는 텍스트 분량만 과금**

---

## 4. UI/UX 처리 원칙

### 입력 단계

* 크레딧 예상 사용량을 **실시간으로 표시**
* 110,000자 초과 시:

  * 생성 버튼 클릭 시 즉시 차단
  * 에러 메시지 + 토스트로 명확하게 안내

### 메시지 톤

* ❌ "70,000자 초과 시 2크레딧" 같은 딱딱한 규칙 나열
* ✅ "대부분의 콘텐츠는 1크레딧만 사용돼요"
* 큰 분량만 예외적으로 2크레딧을 쓰는 구조로 인식 유도

---

## 5. 랜딩 페이지 문구 전략

### 요약 메시지

* **대부분의 콘텐츠는 1크레딧만 사용돼요**
* 아주 큰 분량의 텍스트만 2크레딧이 필요해요

### 상세 보기 (toggle)

* 70,000자 이하 → 1 크레딧
* 70,001 ~ 110,000자 → 2 크레딧
* 110,000자 초과 → 처리 불가

---

## 6. 다국어(i18n) 적용

* 기존 `creditRule` 구조를 새로운 정책에 맞게 재정의
* 한국어/영어 모두 동일한 의미 구조 유지
* 숫자/범위는 메시지 내부에 자연스럽게 포함

---

## 7. ChatGPT vs Cursor – 협업 방식에 대한 결론

### ChatGPT가 더 좋았던 부분

* 요구사항이 계속 변하는 상황

  * 6,000자 → 70k / 110k
  * 초과 요청 거절
  * 정규화 기준 추가
  * UI 문구 톤 변경
* **말로 합의하며 설계 정리하는 단계**

### Cursor가 더 좋은 부분

* 요구사항이 확정된 이후
* 실제 코드 반영, 리팩토링, 타입 체크

### 한 줄 정리

> **설계와 합의는 ChatGPT, 구현과 수정은 Cursor**

---

## 8. 개인적인 메모

* 코드보다 먼저 **정책과 메시지를 정리하는 게 훨씬 빠르다**
* 사용자가 이해하는 언어로 설명하는 것이 과금 UX의 핵심
* 이 과정을 영상으로 남긴 건, 나중에 다시 봐도 좋은 자산이 될 것
