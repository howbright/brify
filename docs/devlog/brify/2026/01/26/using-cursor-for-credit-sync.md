# Cursor를 활용한 크레딧 연동 및 풀스택 워크스페이스 구성

본 문서는 Cursor를 실제 개발 환경에 적용하여  
프론트엔드와 백엔드를 연결하고, 크레딧 연동 작업을 수행한 과정을 정리한 기록입니다.  
단순한 기능 구현이 아니라, Cursor를 개발 워크플로우에 편입시키는 데 목적을 두고 진행했습니다.

---

## 오늘 수행한 작업 요약

오늘은 다음과 같은 작업을 진행했습니다.

- Cursor Pro 환경을 설정하고 실제 사용을 시작했습니다.
- `brify`와 `brify-backend` 레포지토리를 하나의 워크스페이스로 구성했습니다.
- Ask와 Agent의 역할을 구분하여 사용하는 작업 흐름을 정리했습니다.
- 크레딧 조회 API를 프론트엔드와 백엔드에 연동했습니다.
- 코드 변경 사항을 diff로 검토한 후 적용했습니다.
- 이 전체 과정을 영상으로 녹화했습니다.

---

## Cursor를 도입하게 된 배경

기존 작업 방식에서는 다음과 같은 흐름이 반복되었습니다.

- ChatGPT 웹에서 질문
- 답변 복사
- VS Code에 붙여넣기
- 코드 수정 및 재확인

Cursor를 사용하면서부터는 코드 맥락을 유지한 상태에서  
질문, 설계, 수정이 하나의 IDE 안에서 자연스럽게 이어졌습니다.

특히 Ask를 통해 구조를 정리하고,  
Agent가 실제 변경 사항을 diff 형태로 먼저 제안하는 흐름이 인상적이었습니다.

---

## 핵심 작업 1  
### 프론트엔드와 백엔드 워크스페이스 통합

`brify`와 `brify-backend` 두 개의 레포지토리를  
Cursor의 하나의 워크스페이스로 구성했습니다.

이 구성을 통해 다음과 같은 효과를 확인했습니다.

- 백엔드 API 구조를 즉시 탐색할 수 있었습니다.
- 동일한 맥락을 유지한 채 프론트엔드 연동 작업을 진행할 수 있었습니다.
- 풀스택 작업 시 전환 비용이 눈에 띄게 줄어들었습니다.

1인 풀스택 개발 환경에서 특히 효과적인 방식이라고 판단했습니다.

---

## 핵심 작업 2  
### 크레딧 조회 연동

이미 존재하던 크레딧 조회 API를 확인한 후,  
프론트엔드에서 `currentCredits`를 props로 전달하는 구조로 연결했습니다.

ScriptInputCard 컴포넌트와 연동하여 UI에 반영했으며,  
Agent가 변경 사항을 diff 형태로 먼저 제시해 주어  
검토 후 안정적으로 적용할 수 있었습니다.

이 과정에서 별도의 `git diff` 없이도  
PR 리뷰에 가까운 흐름으로 작업이 가능했습니다.

---

## Ask와 Agent 사용 방식 정리

오늘 작업을 통해 Ask와 Agent의 역할이 명확해졌습니다.

- **Ask**
  - 코드 구조 파악
  - API 탐색
  - 설계 및 영향 범위 검토

- **Agent**
  - 실제 코드 수정
  - 여러 파일 변경
  - import 및 연결 작업

질문과 설계는 Ask에서,  
적용은 Agent에서 수행하는 방식이 가장 안정적이라고 판단했습니다.

## 정리

Cursor는 코드를 대신 작성해 주는 도구라기보다,  
생각, 설계, 적용 사이의 병목을 줄여주는 개발 환경이라는 인상을 받았습니다.  
앞으로도 실제 서비스 개발에 지속적으로 활용할 예정입니다.
